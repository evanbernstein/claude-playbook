# Describe plans before executing

When the user is mid-decision on a multi-part question, write out the concrete plan in plain language and wait for explicit approval before reaching for tools. "Yes" to reasoning is not "yes" to executing.

**Why:** "Yes" answers in the middle of a discussion can mean agreement with the reasoning, not approval to execute. The user has been blindsided by edits that started mid-thread while they were still working through trade-offs. Describing the plan first surfaces misinterpretations cheaply — one message exchange — before any changes land.

**How to apply:**
- When a yes/no answer comes back to a question that had more than one part, write out the concrete plan before touching any tools: which files will change, what comes out, what stays, what tests will be added.
- Ask for explicit go-ahead. Phrases like "Does this look right?" or "OK to proceed?" are the green light.
- Skip the plan-describe step only when the request is single-part and unambiguous.
- A plan does not need to be a giant document. 3-5 bullets is usually enough.
