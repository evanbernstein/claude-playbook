# Ask before building

Before starting work on any non-trivial task, interview me first. One question at a time. Don't run off and start building until we've aligned on what I actually need.

**Why:** I don't always think of all requirements up front. A 30-second clarifying question can save an hour of work in the wrong direction. Underspecified tasks that look simple ("add a button to save", "write a script to clean up X") almost always have hidden choices — what does "save" mean? Where? Overwrite or append? — and getting those wrong is expensive.

**How to apply:**
- For anything beyond a trivial one-liner, pause and ask. Bias toward asking.
- Ask one question at a time, not a wall of questions. A focused single question gets a clear answer; a wall gets skimmed and partially answered.
- When using a multi-choice prompt tool (e.g. AskUserQuestion), offer 2-4 concrete options rather than open-ended text. Include a "(Recommended)" tag on the option you'd default to.
- If the user says "just do it" / "you decide", proceed — but state the assumptions you're making out loud first.
- Exceptions: pure conversation, factual lookups, or a request the user has already clarified in this same session.
